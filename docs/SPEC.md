# 项目 SPEC - Lightweight Cross-Platform Desktop Project Editor

## 最终目标
将当前 Flutter Desktop App 演进为轻量、高性能、跨平台的本地项目编辑器(轻量 flutter 版 vscode)，优先支持 macOS 和 Windows，后续可平滑扩展到 Linux。

目标体验接近 VS Code 的本地项目主路径，但坚持更轻、更快、更高信息密度：
- 打开本地文件夹后直接进入项目编辑窗口
- 文件树浏览、代码/文本编辑、多标签页、基础查找与保存
- 每个项目使用独立窗口，而不是把所有内容塞进一个笨重的超级工作台
- 尽可能复用现有 Flutter UI、工作区树、编辑器与同步逻辑

## 核心场景
1. 用户把本地文件夹或文件拖到 App 图标上，或从应用内选择本地路径。
2. App 直接打开一个独立项目窗口，而不是打断到主应用流程。
3. 窗口进入本地项目编辑模式，展示文件树、标签页和编辑区。
4. 用户在轻量工作台中浏览、编辑、保存本地文件；若项目已绑定云端，可继续复用现有混合同步能力。

## 平台范围
- P0: macOS, Windows
- P1: Linux
- P2: 更深的系统集成，例如文件关联、最近项目、Jump List / Dock 菜单增强

## 设计原则
- 轻量优先：不做大而全 IDE，不引入调试器、终端、LSP 全家桶。
- 高性能优先：冷启动快、窗口打开快、文件树扫描和编辑响应稳定。
- 高信息密度：桌面端布局紧凑，减少卡片化 UI。
- 复用优先：优先复用现有 `ProjectChatCodeTab`、本地工作区模型、同步状态、最近工作区逻辑。
- 平台差异隔离：系统级打开入口与窗口管理在平台层隔离，Flutter 编辑体验尽量统一。

## 非目标
- 完整 IDE 替代
- 内置终端 / 调试器 / 断点系统
- 重型 Git GUI
- 海量插件生态兼容层

## 里程碑拆分
### M1. 跨平台多窗口基础架构
- 统一桌面窗口启动上下文
- 主应用与“本地项目窗口”分离
- 独立项目窗口支持欢迎页和最近工作区
- macOS 复用现有原生多窗口链路
- Windows 支持通过命令行参数/新进程直接打开项目窗口

### M2. 原生拖拽与系统入口
- macOS Dock / open-document 稳定打开独立窗口
- Windows 图标/任务栏拖放与参数传递
- 平台通道或 runner 层统一把路径送入 Flutter

### M3. 高密度文件浏览器
- 优化树形浏览
- 控制大目录扫描成本
- 提高桌面端密度和可读性

### M4. 轻量代码编辑器体验
- 强化标签页、多缓冲区、脏状态、保存体验
- 按需补齐查找、折叠、快捷键
- 保持编辑器内核轻量

## 当前实现策略
- 不推倒现有 macOS 本地工作区能力。
- 先补统一桌面启动模型，再把 macOS 和 Windows 接入同一条 `/local-workspace` 路由。
- 本地项目窗口默认使用轻量欢迎页，打开路径后进入已存在的高密度工作区 UI。
- 继续沿用现有本地工作区读写服务，并扩展到 Windows。

## 验证要求
- 每个里程碑结束至少执行：
  - `flutter analyze`
  - `flutter build macos`
  - `flutter build windows`（需在 Windows 主机验证）
- 任何无法在当前环境完成的验证，必须显式记录在 `docs/PLAN.md` 的 `BLOCKERS` 或验证说明中。
