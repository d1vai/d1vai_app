## 🏗️ 架构改进

### 1. 完整的认证流程
```
启动应用
    ↓
检查认证状态
    ↓
┌─────────────────────────────┐
│  已登录                      │
│  ┌─────────────────────────┐ │
│  │ 需要 onboarding?         │ │
│  ├─────────────────────────┤ │
│  │ YES → 显示 Onboarding   │ │
│  │ NO  → 跳转 Dashboard    │ │
│  └─────────────────────────┘ │
└─────────────────────────────┘
            │
            ▼
┌─────────────────────────────┐
│  未登录                      │
│  → 跳转 Login 页面           │
└─────────────────────────────┘
```

### 2. Onboarding 数据流
```
步骤 1: 邀请码
  ↓
authProvider.acceptInvitation(code)
  ↓
保存到本地存储
  ↓
步骤 2: 公司信息
  ↓
authProvider.saveCompanyInfo()
  ↓
保存到本地存储 + 更新用户信息
  ↓
步骤 3: 头像选择
  ↓
authProvider.uploadAvatar() / generateAiAvatars()
  ↓
更新用户信息
  ↓
步骤 4: 完成
  ↓
authProvider.completeOnboarding()
  ↓
清理本地存储 → 跳转到 Dashboard
```

## 💡 关键决策记录

### 1. 为什么选择 Dialog 而不是全屏显示 Onboarding？
- **优点**: 不干扰主应用导航，用户可随时关闭
- **缺点**: 可能被系统错误清除
- **决定**: 使用 Dialog，在完成后清理数据，避免状态不一致

### 2. 为什么使用 DiceBear 作为头像生成服务？
- **优点**: 免费、无需 API 密钥、快速生成、SVG 格式可缩放
- **缺点**: 风格相对简单
- **决定**: 足够满足 MVP 需求，后续可替换为更高级的服务

### 3. 为什么使用 SharedPreferences 而不是 Hive？
- **优点**: 简单、稳定、无需额外序列化代码
- **缺点**: 仅支持基本数据类型
- **决定**: Onboarding 数据量小，SharedPreferences 足够
